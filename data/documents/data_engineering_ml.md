# Data Engineering for Machine Learning

Data engineering for ML encompasses the practices, tools, and architectures for collecting, storing, processing, and serving data for machine learning pipelines. High-quality data engineering is often the difference between a successful and failed ML project.

## Data Collection

### Sources

- **Structured databases**: SQL databases (PostgreSQL, MySQL), data warehouses (Snowflake, BigQuery, Redshift)
- **Unstructured data**: Text documents, images, audio, video files
- **APIs**: REST APIs, GraphQL endpoints, webhooks
- **Web scraping**: Extracting data from web pages (BeautifulSoup, Scrapy, Selenium)
- **Streaming sources**: Kafka, Kinesis, Pub/Sub for real-time data
- **Public datasets**: Hugging Face Datasets, Kaggle, UCI ML Repository, Common Crawl
- **Synthetic data**: Generated by models or simulation for augmenting training data

### Web Scraping for ML Data

Web scraping is commonly used to build training datasets. Best practices:
- Respect robots.txt and rate limits
- Handle dynamic content (JavaScript-rendered pages) with headless browsers
- Implement error handling and retry logic
- Clean HTML to extract meaningful text (remove navigation, ads, boilerplate)
- Deduplicate scraped content
- Maintain data provenance (record source URLs, timestamps)

## Data Preprocessing

### Text Preprocessing

Common text preprocessing steps:
- **Tokenization**: Splitting text into tokens (words, subwords, characters)
- **Lowercasing**: Standardizing case (not always appropriate — "US" vs "us")
- **Stopword removal**: Removing common words (for BM25 search, not for embeddings)
- **Stemming/Lemmatization**: Reducing words to root forms
- **Unicode normalization**: Standardizing character encodings
- **Special character handling**: Removing or replacing HTML entities, control characters

### Subword Tokenization

Modern LLMs use subword tokenizers that handle any text:
- **BPE (Byte Pair Encoding)**: Iteratively merges frequent character pairs. Used by GPT-2/3/4.
- **WordPiece**: Similar to BPE but scores merges by likelihood. Used by BERT.
- **SentencePiece**: Language-agnostic tokenizer treating input as raw Unicode. Used by Llama, T5.
- **tiktoken**: OpenAI's fast BPE tokenizer. cl100k_base is used for GPT-4 and ChatGPT.

Tokenizer vocabulary sizes: GPT-4 uses ~100K tokens, Llama uses ~32K (Llama 3 uses 128K).

### Data Cleaning

- **Missing values**: Imputation (mean, median, mode, KNN, model-based) or removal
- **Outliers**: Detection (IQR, z-score, isolation forest) and handling
- **Duplicates**: Exact and near-duplicate detection (MinHash, SimHash)
- **Inconsistencies**: Standardizing formats (dates, units, categories)
- **Label noise**: Identifying and correcting mislabeled examples (Cleanlab)

### Feature Engineering

Creating informative features from raw data:
- **Numerical**: Scaling (StandardScaler, MinMaxScaler), log transforms, polynomial features, binning
- **Categorical**: One-hot encoding, label encoding, target encoding, hash encoding
- **Text**: TF-IDF, embeddings, n-grams, regex features
- **Temporal**: Day of week, hour, lag features, rolling statistics, seasonality decomposition
- **Interaction features**: Combining multiple features (ratios, products, differences)

## Data Quality

### Data Validation

Automated checks to catch data quality issues early:
- **Schema validation**: Check data types, column names, value ranges
- **Statistical validation**: Compare distributions against historical baselines
- **Completeness checks**: Monitor null rates per column
- **Freshness checks**: Ensure data is up-to-date

Tools: Great Expectations, Pandera, Pydantic, Cerberus.

### Data Versioning

Track changes to datasets like code:
- **DVC (Data Version Control)**: Git-like tool for versioning data and ML models. Stores metadata in Git, data in cloud storage (S3, GCS, Azure).
- **LakeFS**: Git-like branching for data lakes
- **Delta Lake**: ACID transactions on data lakes, time travel for data versioning

## Document Processing for RAG

### Document Loading

Tools for extracting text from various formats:
- **PyMuPDF (fitz)**: Fast, reliable PDF text extraction
- **pdf2image + Tesseract**: OCR for scanned PDFs
- **python-docx**: Microsoft Word document parsing
- **markdown**: Converting Markdown to plain text
- **BeautifulSoup**: HTML parsing and text extraction
- **Unstructured.io**: Universal document parser supporting 20+ formats

### Chunking Strategies

Splitting documents into appropriately-sized chunks for embedding and retrieval:
- **Fixed-size chunking**: Split every N tokens/characters. Simple but may split mid-sentence.
- **Sentence-based chunking**: Split on sentence boundaries. More natural but variable chunk sizes.
- **Semantic chunking**: Split when embedding similarity between consecutive sentences drops below a threshold

Key parameters:
- **Chunk size**: 200-800 tokens typical. Smaller = more precise retrieval; larger = more context per chunk
- **Overlap**: 50-200 tokens overlap between consecutive chunks prevents losing context at boundaries
- **Metadata preservation**: Track source document, page number, section heading for each chunk

### Text Extraction Quality

Common issues and solutions:
- **PDF layout**: Multi-column layouts, tables, and headers can cause garbled text extraction
- **OCR errors**: Scanned documents require OCR, which introduces character-level errors
- **Encoding issues**: Mixed encodings, special characters, and Unicode normalization
- **Boilerplate removal**: Headers, footers, page numbers, and watermarks should be stripped
- **Table extraction**: Tools like Camelot and Tabula specialize in extracting tables from PDFs

## Data Pipelines

### ETL vs ELT

- **ETL (Extract, Transform, Load)**: Transform data before loading into the destination. Traditional approach.
- **ELT (Extract, Load, Transform)**: Load raw data first, then transform in the destination. Modern approach leveraging cloud data warehouse compute.

### Pipeline Orchestration

Tools for scheduling and managing data pipeline DAGs:
- **Apache Airflow**: The most popular open-source orchestrator. Python-defined DAGs, extensive integrations.
- **Prefect**: Modern alternative to Airflow with simpler API and better error handling.
- **Dagster**: Data-aware orchestrator with built-in data lineage and testing.
- **Luigi**: Spotify's lightweight pipeline framework.

### Batch vs Stream Processing

- **Batch**: Process data in scheduled intervals (hourly, daily). Tools: Spark, Dask, Pandas.
- **Streaming**: Process data in real-time as it arrives. Tools: Apache Kafka, Apache Flink, Spark Structured Streaming.
- **Micro-batch**: Small batches at high frequency (seconds). A practical middle ground.

## Storage

### Vector Databases

Specialized databases for storing and searching embeddings:
- **ChromaDB**: Lightweight, open-source, embedded. Great for prototyping and small-medium workloads.
- **Pinecone**: Managed cloud service. Serverless, easy to scale, no infrastructure management.
- **Weaviate**: Open-source with hybrid search (vector + keyword). GraphQL API.
- **Qdrant**: Open-source, Rust-based, high performance. Supports filtering and payloads.
- **Milvus**: Open-source, designed for billion-scale vector search. GPU acceleration.
- **pgvector**: PostgreSQL extension for vector similarity search. Use existing Postgres infrastructure.

### Data Lakes

Centralized repository for raw data in its native format:
- **Object storage**: S3, GCS, Azure Blob as the foundation
- **File formats**: Parquet (columnar, compressed), Delta Lake (ACID), Iceberg (table format)
- **Query engines**: Trino, Athena, Spark SQL for querying data lakes directly
